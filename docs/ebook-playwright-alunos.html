playwright.config

Configuração Essencial
O arquivo playwright.config.ts
  • Centraliza todas as configurações globais do Playwright
  • Define browsers, timeouts, baseURL, retries e paralelismo
  • Controla a geração automática de traces, vídeos e screenshots
  • Permite configurar diferentes projects (browsers, devices)
  • Pode ser sobrescrito via CLI ou por teste individual
  • Documentação completa: playwright.dev/docs/test-configuration
Artefatos de debug automáticos
  • trace: 'on-first-retry' — grava trace quando o teste falha e é re-executado
  • trace: 'on' — grava trace em toda execução (útil em CI)
  • video: 'on-first-retry' — grava vídeo do browser quando há retry
  • screenshot: 'only-on-failure' — captura screenshot automaticamente ao falhar
  • Artefatos ficam na pasta test-results/
  • O Trace Viewer é muito mais completo que screenshots ou vídeos — permite inspecionar o DOM, ver a timeline e até fazer pick locator. Acesse via trace.playwright.dev
Retries, Workers e Paralelismo
  • retries — número de vezes que um teste falho será re-executado. Recomendado: 1 (máximo 2 para projetos novos)
  • Quanto mais retries, mais tempo e recurso consumido — se o teste precisa de 3 retries para passar, investigue a causa raiz
  • workers — número de threads paralelas. Depende do hardware e da infra de CI
  • fullyParallel: true — roda testes individuais em paralelo (não apenas arquivos)
  • Use test.describe.serial quando precisar que testes rodem em sequência dentro de um arquivo
  • Dica: use ternários para configurar valores diferentes em CI vs local — ex.: retries: process.env.CI ? 2 : 0
forbidOnly, Reporters e Timeouts
  • forbidOnly — impede que .only suba para a pipeline. Essencial em CI para evitar que apenas um teste rode por engano
  • Reporters disponíveis: html, dot, list, line, json, junit — o junit é ideal para integração com GitHub/GitLab, mostrando resultados direto no PR. Veja: playwright.dev/docs/test-reporters
  • Dois tipos de timeout: timeout de ação (para cliques, navegação) e timeout de expect (para assertions)
  • Prefira configurar timeout por ação específica em vez de aumentar o timeout global — ex.: await expect(el).toBeVisible({ timeout: 15000 })
  • baseURL pode ser uma string fixa ou uma função que retorna a URL dinâmica do ambiente (útil para feature branches)
// playwright.config.ts — configuração completa com artefatos e boas práticas
import { defineConfig } from '@playwright/test';

export default defineConfig({
  testDir: './tests',
  fullyParallel: true,
  forbidOnly: !!process.env.CI,
  retries: process.env.CI ? 2 : 0,
  workers: process.env.CI ? 4 : undefined,
  reporter: 'html',

  use: {
    baseURL: 'https://exemplo.com',
    trace: 'on-first-retry',
    video: 'on-first-retry',
    screenshot: 'only-on-failure',
  },

  projects: [
    { name: 'chromium', use: { browserName: 'chromium' } },
    { name: 'firefox', use: { browserName: 'firefox' } },
  ],
});
// Timeout específico por ação (em vez de aumentar o global)
await expect(page.getByText('Dados carregados')).toBeVisible({ timeout: 15000 });

// test.describe.serial — forçar execução sequencial em um arquivo
test.describe.serial('fluxo de checkout', () => {
  test('adicionar produto', async ({ page }) => { /* ... */ });
  test('preencher endereço', async ({ page }) => { /* ... */ });
  test('finalizar compra', async ({ page }) => { /* ... */ });
});

// test.skip e test.only — controlar quais testes rodam
test.skip('feature em desenvolvimento', async ({ page }) => { /* ... */ });
test.only('debug temporário', async ({ page }) => { /* ... */ });
💡 Para lembrar

Use trace: 'on-first-retry' como padrão — você ganha visibilidade total quando um teste falha sem impactar a performance das execuções que passam. Em CI, considere trace: 'on' para ter sempre o diagnóstico completo. O Trace Viewer mostra o DOM completo, timeline, network e permite inspecionar elementos — na maioria dos casos é muito mais útil que screenshots ou vídeos.

⚠️ Cuidado com .only

É muito comum esquecer um test.only() no código e comitar para a pipeline. Quando isso acontece, só aquele teste roda e os outros são ignorados — dando uma falsa impressão de que tudo passou. Use forbidOnly: !!process.env.CI no config para evitar esse problema.

Organização e Arquitetura

Arquitetura Boas Práticas
Page Object Model (POM)
  • Padrão que encapsula os locators e ações de cada página em uma classe
  • Separa a lógica de interação da lógica de teste
  • Facilita a manutenção: se a UI muda, você atualiza só o Page Object
  • Cada página da aplicação vira uma classe com métodos reutilizáveis
  • Testes ficam mais legíveis e curtos
Boas práticas de Page Object
  • Nomeie sempre a classe com o sufixo Page — ex.: LoginPage, CheckoutPage — facilita o autocomplete e evita duplicidade
  • Nomeie variáveis de locator com o tipo do elemento: submitButton, emailInput, errorLabel
  • Coloque métodos de assert no Page Object (prefixe com assert) para reutilizar em vários specs
  • Mantenha arquivos com no máximo ~200 linhas — refatore quando crescer demais
  • Comece simples e divida à medida que o projeto cresce — não tente antecipar toda a estrutura
  • Documentação: playwright.dev/docs/pom
// pages/login.page.ts — Page Object com padrões de nomenclatura
import { type Page, type Locator, expect } from '@playwright/test';

export class LoginPage {
  private emailInput: Locator;
  private passwordInput: Locator;
  private submitButton: Locator;
  private errorLabel: Locator;

  constructor(private page: Page) {
    this.emailInput = page.getByLabel('E-mail');
    this.passwordInput = page.getByLabel('Senha');
    this.submitButton = page.getByRole('button', { name: 'Entrar' });
    this.errorLabel = page.getByTestId('login-error');
  }

  async goto() {
    await this.page.goto('/login');
  }

  async fillUsername(email: string) {
    await this.emailInput.fill(email);
  }

  async login(email: string, senha: string) {
    await this.emailInput.fill(email);
    await this.passwordInput.fill(senha);
    await this.submitButton.click();
  }

  async assertLoginSuccess() {
    await expect(this.page).toHaveURL(/dashboard/);
  }

  async assertErrorVisible(message: string) {
    await expect(this.errorLabel).toContainText(message);
  }
}

// tests/login.spec.ts — Teste usando o Page Object
import { test } from '@playwright/test';
import { LoginPage } from '../pages/login.page';

test('login com sucesso', async ({ page }) => {
  const loginPage = new LoginPage(page);
  await loginPage.goto();
  await loginPage.login('user@email.com', 'senha123');
  await loginPage.assertLoginSuccess();
});

test('exibe erro com senha inválida', async ({ page }) => {
  const loginPage = new LoginPage(page);
  await loginPage.goto();
  await loginPage.login('user@email.com', 'errada');
  await loginPage.assertErrorVisible('Credenciais inválidas');
});
🏗️ Page Object não é só para páginas

Você pode criar Page Objects para componentes reutilizáveis como menus, modais ou barras de navegação — ex.: TopMenuPage com o método logout() que aparece em todas as páginas. Também pode criar "Request Objects" para chamadas de API que encapsulam URL, método e payload.

🏠 Para praticar em casa
  • Crie um Page Object para uma página de cadastro com métodos goto() e register()
  • Refatore um teste existente para usar o Page Object e compare a legibilidade
  • Crie um Page Object para um componente (ex.: menu de navegação) e importe-o em dois specs diferentes

Fixtures

Arquitetura Performance
O que são Fixtures
  • Fixtures são um mecanismo nativo do Playwright para compartilhar setup entre testes e injetar dependências automaticamente
  • Substituem beforeEach repetitivos — o código roda antes do teste e é limpo automaticamente
  • Gerenciam o ciclo de vida de recursos: page objects, dados, contextos de browser
  • Podem acelerar a execução em até 70% ao manter objetos na memória entre testes
  • Separação clara: fixtures para setup, page objects para ações, specs para validações
  • Documentação: playwright.dev/docs/test-fixtures
Quando usar (e quando não usar)
  • Use quando: muitos page objects, suíte grande, padrão repetitivo entre specs, dependências claras
  • Não use em projetos pequenos onde cada teste é diferente — adiciona complexidade desnecessária
  • Vantagem: elimina linhas de new PageObject() e beforeEach de todos os specs
  • Desvantagem: curva de aprendizado, pode carregar coisas na memória desnecessariamente, e código "morto" pode acumular
  • Se os fluxos do sistema são muito independentes, prefira uma fixture por page object em vez de uma fixture única
  • Mantenha regras de lint e revisões para evitar code debt nas fixtures
// fixtures/auth.fixture.ts — Fixture customizada
import { test as base } from '@playwright/test';
import { LoginPage } from '../pages/login.page';
import { DashboardPage } from '../pages/dashboard.page';

type MyFixtures = {
  loginPage: LoginPage;
  dashboardPage: DashboardPage;
};

export const test = base.extend<MyFixtures>({
  loginPage: async ({ page }, use) => {
    const loginPage = new LoginPage(page);
    await loginPage.goto();
    await use(loginPage);
  },

  dashboardPage: async ({ page }, use) => {
    await use(new DashboardPage(page));
  },
});

export { expect } from '@playwright/test';

// tests/login.spec.ts — Usando a fixture (import do fixture, não do playwright)
import { test, expect } from '../fixtures/auth.fixture';

test('login com sucesso', async ({ loginPage }) => {
  await loginPage.login('user@email.com', 'senha123');
  await loginPage.assertLoginSuccess();
});

// tests/dashboard.spec.ts — Reutilizando a mesma fixture
import { test, expect } from '../fixtures/auth.fixture';

test('dashboard exibe dados do usuário', async ({ dashboardPage }) => {
  // dashboardPage já está pronto — fixture fez o setup
  await expect(dashboardPage.welcomeLabel).toBeVisible();
});
💡 Como a Fixture funciona

A fixture executa tudo que está antes da chamada use() como setup, passa o objeto para o teste, e executa tudo que está depois do use() como teardown. O Playwright gerencia isso de forma otimizada — os objetos ficam na memória e são reutilizados, diferente de um beforeEach que recria tudo a cada teste.

⚠️ Cuidado com fixtures grandes

Uma fixture "base" que instancia todos os page objects de uma vez é conveniente, mas carrega tudo na memória mesmo quando o teste precisa de apenas um. Se os fluxos do sistema são independentes, considere criar uma fixture por área — ex.: auth.fixture.ts, checkout.fixture.ts, messages.fixture.ts.

Autenticação

Prática Otimização
storageState — login reutilizável
  • Permite fazer login uma única vez e reutilizar a sessão em todos os testes
  • Salva cookies e localStorage em um arquivo JSON (o arquivo é gerado automaticamente)
  • Cada teste começa já autenticado — muito mais rápido
  • Configurado via project setup no playwright.config.ts
  • Ideal para suítes grandes que precisam de usuário logado
  • O JSON armazena apenas cookies e origins (localStorage) — não serve para dados de carrinho, produto, etc.
  • Documentação: playwright.dev/docs/auth
Duas formas de fazer login no setup
  • Login via UI: simula o fluxo real — navega até a página, preenche campos, clica no botão. Mais lento, mas testa o fluxo completo
  • Login via API: chama o endpoint de autenticação diretamente — muito mais rápido. Use request.newContext() para fazer a chamada
  • Ambas as formas salvam o estado com storageState() e o resultado é idêntico para os testes seguintes
  • Em projetos reais, login via API é o mais recomendado por ser mais rápido e estável
  • Configure via projects com dependencies (abordagem moderna) em vez de globalSetup (legado)
// playwright.config.ts — setup de autenticação com projects (abordagem moderna)
import { defineConfig } from '@playwright/test';

export default defineConfig({
  projects: [
    {
      name: 'setup',
      testDir: './tests/setup',
      testMatch: '**/*.setup.ts',
    },
    {
      name: 'chromium',
      use: {
        browserName: 'chromium',
        storageState: 'playwright/.auth/session.json',
      },
      dependencies: ['setup'],
    },
  ],
});
// tests/setup/auth.setup.ts — Login via UI e salvar estado
import { test as setup } from '@playwright/test';

const authFile = 'playwright/.auth/session.json';

setup('fazer login', async ({ page }) => {
  await page.goto('https://exemplo.com/login');
  await page.getByLabel('E-mail').fill('user@email.com');
  await page.getByLabel('Senha').fill('senha123');
  await page.getByRole('button', { name: 'Entrar' }).click();
  await page.context().storageState({ path: authFile });
});
// tests/setup/auth.setup.ts — Login via API (mais rápido)
import { test as setup, request } from '@playwright/test';

const authFile = 'playwright/.auth/session.json';

setup('fazer login via API', async ({ }) => {
  const apiContext = await request.newContext();
  const response = await apiContext.post('https://exemplo.com/api/login', {
    data: {
      email: 'user@email.com',
      password: 'senha123',
    },
  });
  if (!response.ok()) throw new Error('Falha ao fazer login via API');
  await apiContext.storageState({ path: authFile });
  await apiContext.dispose();
});
Múltiplos perfis (admin + user)
  • Crie um setup separado e um storageState para cada perfil de usuário
  • Configure projects independentes — ex.: 'chromium-admin' e 'chromium-user'
  • Cada spec importa o perfil correto via project ou fixture
  • Útil para: testar permissões, admin vs. usuário regular, fluxos de aprovação
  • Documentação: playwright.dev/docs/auth#multiple-signed-in-roles
Limitações do storageState
  • Armazena apenas cookies e localStorage — dados de sessão server-side não são capturados
  • Não serve para salvar estado de carrinho de compras, dados de formulário ou fluxos multi-step
  • Para pular etapas de um fluxo (ex.: ir direto para checkout), use chamadas de API para preparar os dados
  • O JSON tem data de expiração dos cookies — se expirar, o setup precisa rodar novamente
// playwright.config.ts — múltiplos perfis de autenticação
export default defineConfig({
  projects: [
    { name: 'setup-admin', testMatch: '**/admin.setup.ts' },
    { name: 'setup-user', testMatch: '**/user.setup.ts' },
    {
      name: 'admin-tests',
      use: { storageState: 'playwright/.auth/admin.json' },
      dependencies: ['setup-admin'],
    },
    {
      name: 'user-tests',
      use: { storageState: 'playwright/.auth/user.json' },
      dependencies: ['setup-user'],
    },
  ],
});
⚠️ Cuidado

Adicione playwright/.auth/ ao .gitignore — os JSONs contêm tokens de sessão. Nunca commite arquivos de estado de autenticação no repositório.

Testes Mais Estáveis

Estabilidade Boas Práticas
Esperas corretas e auto-waiting
  • O Playwright já implementa auto-waiting na maioria das ações — consulte a tabela de auto-waiting para ver quais checks cada ação faz
  • Nunca use waitForTimeout() — ele faz um "pause" fixo e torna o teste lento e instável
  • Use assertions com auto-retry (expect) — elas revalidam até o timeout configurado
  • waitForLoadState('networkidle') — espera as requisições de rede terminarem
  • waitForResponse(url) — espera uma resposta específica da API
  • waitForURL(url) — espera a navegação chegar em uma URL
  • Alguns métodos como fill() não esperam por todos os events (ex.: stable) — isso é intencional. Se precisar, espere outro elemento antes
  • Documentação: playwright.dev/docs/actionability
Interceptação de requests (mocks e route)
  • page.route() permite interceptar e modificar requisições de rede
  • route.fulfill() — retorna dados mockados para o browser
  • route.abort() — bloqueia uma requisição (útil para bloquear analytics, imagens, etc.)
  • route.continue() — deixa a requisição prosseguir (opcionalmente com headers modificados)
  • Torna os testes independentes do backend — simule erros, respostas lentas ou dados específicos
  • Pode criar uma classe NetworkInterceptor para gerenciar interceptações de forma reutilizável
  • Documentação: playwright.dev/docs/mock | playwright.dev/docs/network
// Esperar network idle antes de fazer assertions
await page.goto('/dashboard');
await page.waitForLoadState('networkidle');

// Esperar uma resposta específica da API
const response = await page.waitForResponse('**/api/users');
expect(response.status()).toBe(200);

// Esperar navegação para URL específica
await page.waitForURL('**/dashboard');

// Aguardar resposta da API antes de fazer assertion no DOM
const [resp] = await Promise.all([
  page.waitForResponse('**/api/products'),
  page.getByRole('button', { name: 'Carregar' }).click(),
]);

// Mock de uma API — retornar dados fixos
await page.route('**/api/products', async (route) => {
  await route.fulfill({
    status: 200,
    contentType: 'application/json',
    body: JSON.stringify([{ id: 1, name: 'Produto Mock' }]),
  });
});

// Mock de erro — simular falha na API
await page.route('**/api/products', async (route) => {
  await route.fulfill({ status: 500 });
});

// Bloquear requisições desnecessárias (analytics, imagens, etc.)
await page.route('**/*.{png,jpg,jpeg}', (route) => route.abort());
await page.route('**/analytics/**', (route) => route.abort());
💡 Para lembrar

Mocks são ferramentas poderosas, mas use com moderação. Testes com mock validam o frontend isoladamente. Mantenha também testes E2E sem mock para garantir a integração real com o backend. Você pode criar uma classe NetworkInterceptor reutilizável para centralizar as interceptações mais comuns do projeto.

🏠 Para praticar em casa
  • Crie um teste que intercepta uma API e retorna dados mockados — valide que a UI exibe os dados corretos
  • Simule um erro 500 na API e verifique se a mensagem de erro é exibida na tela
  • Use waitForResponse() para esperar uma chamada API antes de fazer assertions

Múltiplos Contextos e Cenários

Avançado Prática
Multi-user — dois usuários no mesmo teste
  • Crie múltiplos browser contexts no mesmo teste
  • Cada contexto tem seus próprios cookies, sessão e storage
  • Ideal para testar: chat em tempo real, permissões, colaboração
  • Ambos os usuários rodam no mesmo browser (sem custo extra)
  • Use browser.newContext() para criar contextos independentes
Multi-tab e popups
  • O Playwright intercepta novas abas e popups automaticamente
  • Use page.waitForEvent('popup') para capturar a nova aba
  • Cada aba é uma Page independente — mesmos métodos de interação
  • Útil para: OAuth, links que abrem em nova aba, janelas de pagamento
  • Não precisa de window handle como em Selenium
// Multi-user — dois usuários simultâneos
const context1 = await browser.newContext();
const context2 = await browser.newContext();
const userA = await context1.newPage();
const userB = await context2.newPage();

await userA.goto('/chat');
await userB.goto('/chat');
await userA.getByLabel('Mensagem').fill('Olá!');
await userA.getByRole('button', { name: 'Enviar' }).click();
await expect(userB.getByText('Olá!')).toBeVisible();

// Multi-tab — capturar popup
const [popup] = await Promise.all([
  page.waitForEvent('popup'),
  page.getByText('Abrir em nova aba').click(),
]);
await expect(popup).toHaveURL(/nova-pagina/);
🏠 Para praticar em casa
  • Crie um teste com dois contextos simulando dois usuários acessando a mesma página
  • Teste um link que abre em nova aba e valide o conteúdo da popup

Mobile e Emulação

Prática Devices
Device presets
  • O Playwright traz presets prontos para dezenas de dispositivos
  • Emula viewport, user-agent, touch, device scale factor
  • Dispositivos comuns: iPhone 14, Pixel 7, iPad, Galaxy S9+
  • Configurável por projeto no playwright.config.ts
  • Lista completa: playwright.dev/docs/emulation#devices
Outras emulações
  • locale — idioma e formatação de datas/moeda
  • geolocation — simular localização GPS
  • permissions — conceder permissões (câmera, microfone, geolocalização)
  • colorScheme — testar dark mode e light mode
  • timezoneId — simular fuso horário diferente
// playwright.config.ts — projeto mobile
import { defineConfig, devices } from '@playwright/test';

export default defineConfig({
  projects: [
    { name: 'Desktop Chrome', use: { ...devices['Desktop Chrome'] } },
    { name: 'Mobile Safari', use: { ...devices['iPhone 14'] } },
    { name: 'Mobile Android', use: { ...devices['Pixel 7'] } },
  ],
});

// Emulação inline no teste
const context = await browser.newContext({
  ...devices['iPhone 14'],
  locale: 'pt-BR',
  geolocation: { latitude: -23.55, longitude: -46.63 },
  permissions: ['geolocation'],
  colorScheme: 'dark',
});
💡 Para lembrar

A emulação mobile do Playwright não é um dispositivo real — ela simula as características do device (viewport, user-agent, touch) no Chromium/WebKit. É como usar o DevTools do Chrome no modo responsivo. Para testes de performance ou bugs específicos de hardware, use dispositivos reais, BrowserStack ou farms de dispositivos.

📱 Testes nativos em mobile

O Playwright não roda testes em aplicações mobile nativas (iOS/Android). Para testes mobile nativos, considere frameworks como Robot Framework (que pode integrar com Playwright) ou ferramentas especializadas como Appium. Documentação de emulação: playwright.dev/docs/emulation

CI com GitHub Actions

CI/CD GitHub
Por que rodar testes em CI
  • Garante que cada PR é testada automaticamente
  • Evita que bugs cheguem à branch principal
  • Testes rodam em ambiente limpo e consistente
  • Resultados e artefatos ficam acessíveis no GitHub
  • Feedback rápido para o time de desenvolvimento
  • Tudo que estiver no playwright.config.ts (trace, retries, workers, reporters) reflete na pipeline
GitHub Actions — setup
  • O Playwright oferece um workflow pronto na instalação (npm init playwright@latest)
  • Arquivo: .github/workflows/playwright.yml
  • Instala dependências, browsers e roda os testes
  • Upload automático do relatório HTML como artefato
  • Roda em Ubuntu por padrão (mais rápido e barato)
  • Use o reporter junit para ver resultados direto no PR do GitHub
  • Documentação: playwright.dev/docs/ci-intro
# .github/workflows/playwright.yml
name: Playwright Tests
on:
  push:
    branches: [ main ]
  pull_request:
    branches: [ main ]

jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: 20
      - name: Install dependencies
        run: npm ci
      - name: Install Playwright Browsers
        run: npx playwright install --with-deps
      - name: Run Playwright tests
        run: npx playwright test
      - uses: actions/upload-artifact@v4
        if: ${{ !cancelled() }}
        with:
          name: playwright-report
          path: playwright-report/
          retention-days: 30
Outras ferramentas de CI
  • GitLab CI: usa .gitlab-ci.yml com stages e needs em vez de jobs. Pode precisar configurar workers/CPU via Kubernetes ou runners
  • CircleCI: usa .circleci/config.yml — estrutura similar mas com terminologia própria
  • Jenkins: use Jenkinsfile com pipeline declarativa ou scripted
  • Consulte sempre a documentação da ferramenta + playwright.dev/docs/ci para ver exemplos específicos
Merge de relatórios e dicas avançadas
  • Se a pipeline roda em múltiplos jobs (sharding), use npx playwright merge-reports para combinar os relatórios — veja docs/test-sharding
  • Configure retention-days para controlar por quantos dias os artefatos ficam disponíveis (padrão GitHub: 30 dias)
  • Entenda a infra de CI da sua empresa — o número de workers depende dos recursos alocados (CPU, memória), não apenas da configuração do Playwright
  • Em ambientes com Kubernetes, pode ser necessário configurar cpu requests separadamente para garantir que os workers tenham recursos suficientes
📁 Dica

Use if: ${{ !cancelled() }} no upload do relatório para garantir que ele seja salvo mesmo quando os testes falham. Assim você pode diagnosticar falhas diretamente no GitHub. Combine com o reporter junit para ver resultados inline nos PRs.

Boas Práticas Reais

Boas Práticas Produção
Estrutura de pastas recomendada
  • tests/ — arquivos .spec.ts organizados por funcionalidade
  • pages/ — Page Objects separados por página ou componente
  • fixtures/ — fixtures customizadas reutilizáveis
  • utils/ — helpers, dados de teste, constantes, data factory
  • playwright.config.ts — na raiz do projeto
  • Comece simples e refatore à medida que o projeto cresce — não tente criar toda a estrutura no dia 1
  • Documentação: playwright.dev/docs/best-practices
Evitando flaky tests
  • Nunca usar waitForTimeout() — use assertions com auto-wait
  • Cada teste deve ser independente — não depender de outro teste
  • Use dados únicos por teste para evitar conflitos em paralelo
  • Limpe o estado entre testes (logout, reset de banco, etc.)
  • Monitore testes que falham intermitentemente e corrija a causa raiz
  • Se o banco de dados bloqueia tabelas durante operações (create/update/delete), isso pode causar falhas com muitos workers — ajuste o paralelismo
Regras de lint e qualidade de código
  • Configure ESLint com regras específicas para o projeto de testes — impede que código problemático entre no repositório
  • max-lines: limite cada arquivo a ~200 linhas para manter a legibilidade
  • max-params: limite o número de parâmetros por método
  • complexity: complexidade ciclomática máxima de 10
  • no-restricted-syntax: bloqueie .only em specs para evitar comitar por acidente
  • no-duplicate-else-if, no-dupe-args: evite duplicidade
  • Adicione lint ao pipeline para rejeitar PRs que violem as regras
Data Factory e dados de teste
  • Crie um Data Factory para gerar dados dinâmicos por teste — evita conflitos entre testes paralelos
  • Em vez de escrever vários testes similares, escreva um teste e itere nos dados com test.describe + loop ou parametrização
  • Centralize mensagens de texto e labels em um arquivo de constantes — facilita manutenção quando a UI muda
  • Use variáveis de ambiente para credenciais e URLs de diferentes ambientes
  • Considere usar Faker.js para gerar dados realistas (nomes, e-mails, endereços)

🔄 Atualização do Playwright

Atualize o Playwright regularmente com npm install @playwright/test@latest e depois npx playwright install. Dica: espere pela versão .1 ou .2 de cada release (ex.: 1.59.1 em vez de 1.59.0) para evitar bugs de versão inicial. Acompanhe as releases em playwright.dev/docs/release-notes ou ative notificações no GitHub (Watch > Custom > Releases).

⚠️ Regra de ouro

Se um teste falha intermitentemente (flaky), não ignore. Investigue a causa: pode ser uma espera incorreta, dado compartilhado entre testes, ou um bug real na aplicação. Flaky tests destroem a confiança da suíte inteira.
