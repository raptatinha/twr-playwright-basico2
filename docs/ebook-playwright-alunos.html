O que é Playwright

Conceito Teoria
O que é
  • Biblioteca open-source criada e mantida pela Microsoft
  • Permite automatizar browsers para testes E2E (end-to-end)
  • Suporta Chromium, Firefox e WebKit com uma única API
  • Disponível para JavaScript, TypeScript, Python, Java e .NET
  • Principal diferencial: auto-wait nativo — menos testes frágeis
Quando usar Playwright
  • Testar fluxos completos de usuário: login, checkout, formulários
  • Validar a aplicação em múltiplos browsers simultaneamente
  • Cenários com múltiplas abas, downloads ou uploads
  • Projetos com stack TypeScript ou JavaScript
  • Quando a velocidade e o paralelismo são importantes
💡 Para lembrar

O Playwright testa a sua aplicação da perspectiva do usuário — abre um browser de verdade, clica em botões, preenche formulários e verifica se tudo funciona. É diferente de testes unitários, que testam funções isoladas de código.

Setup e Primeiro Teste

Prática Instalação
Instalando do zero — passo a passo
1
Crie uma pasta vazia e abra no VS Code mkdir meu-projeto && cd meu-projeto && code .
2
Rode o instalador interativo O wizard vai perguntar: TypeScript ou JavaScript, nome da pasta de testes, se quer configurar GitHub Actions. Pode aceitar os padrões.
3
Explore a estrutura gerada Veja os arquivos: playwright.config.ts, tests/, tests-examples/ e package.json
4
Rode os testes de exemplo Use npx playwright test e observe a saída no terminal
5
Abra o relatório HTML Use npx playwright show-report para ver os resultados no browser 
# Instalar Playwright (rode dentro da pasta do projeto)
npm init playwright@latest

# Rodar todos os testes
npx playwright test

# Rodar apenas um arquivo específico
npx playwright test example.spec.ts

# Ver o relatório HTML
npx playwright show-report
Estrutura do projeto
  • playwright.config.ts — configurações globais do Playwright
  • tests/ — onde ficam seus arquivos .spec.ts
  • tests-examples/ — exemplo completo gerado pelo wizard
  • package.json — dependências e scripts do projeto
O primeiro teste gerado
  • Abre playwright.dev no browser
  • Verifica o título da página com toHaveTitle
  • Clica em um link e verifica a URL
  • É um ótimo ponto de partida para entender a estrutura
📁 Dica

O arquivo tests-examples/demo-todo-app.spec.ts tem exemplos mais ricos de tudo que vimos na aula. Vale a pena ler e rodar quando estiver praticando em casa.

Locators

Conceito Boas Práticas
Locators recomendados
  • getByRole() — por papel de acessibilidade (button, link, heading...)
  • getByText() — pelo texto visível do elemento
  • getByLabel() — por label associado ao input
  • getByPlaceholder() — pelo placeholder do input
  • getByTestId() — por atributo data-testid
  • getByAltText() — por atributo alt de imagens
  • getByTitle() — por atributo title
Por que evitar CSS e XPath
  • Quebram facilmente quando o layout muda
  • Não refletem como o usuário real interage com a página
  • Dificultam a manutenção dos testes ao longo do tempo
  • Locators semânticos são mais legíveis e resistentes a mudanças
  • O Playwright prioriza acessibilidade — boa prática para o produto
// ✅ Bom — usa getByRole (semântico e robusto)
await page.getByRole('button', { name: 'Entrar' }).click();

// ✅ Bom — usa getByLabel (ideal para formulários)
await page.getByLabel('E-mail').fill('teste@email.com');

// ✅ Bom — usa getByTestId (quando há data-testid no HTML)
await page.getByTestId('submit-btn').click();

// ❌ Evitar — CSS frágil, quebra com refatorações de UI
await page.locator('.btn.btn-primary.submit > span').click();

// ❌ Evitar — XPath longo e difícil de manter
await page.locator('//div[@class="form"]/button[2]').click();
🏠 Para praticar em casa
  • Rode npx playwright codegen https://playwright.dev e interaja com a página. Veja os locators sendo gerados automaticamente
  • Abra uma página qualquer, inspecione os elementos e tente escrever locators usando getByRole e getByLabel

Interações Básicas

Prática Referência
Ações de mouse e teclado
  • click() — clica no elemento
  • dblclick() — duplo clique
  • hover() — passa o mouse sobre o elemento
  • fill() — preenche um input (limpa o valor anterior)
  • type() — digita caractere por caractere
  • press('Enter') — pressiona uma tecla (Enter, Tab, Escape...)
  • selectOption() — seleciona valor em um dropdown
  • setInputFiles() — faz upload de um arquivo
  • check() / uncheck() — marca/desmarca checkbox
Diferença entre fill() e type()
  • fill() limpa o campo antes de escrever — mais rápido e direto
  • type() simula a digitação real, tecla por tecla — mais lento
  • Na maioria dos casos, prefira fill()
  • Use type() quando o site tiver lógica que depende de eventos de teclado intermediários (ex: autocomplete)
// Preenchendo um formulário de login completo
await page.goto('https://exemplo.com/login');
await page.getByLabel('E-mail').fill('user@email.com');
await page.getByLabel('Senha').fill('senha123');
await page.getByRole('button', { name: 'Entrar' }).click();

// Selecionando uma opção em um dropdown
await page.getByLabel('País').selectOption('Brasil');

// Pressionando Enter após preencher um campo de busca
await page.getByRole('searchbox').press('Enter');
🏠 Para praticar em casa
  • Escreva um teste que acesse playwright.dev, clique em "Docs" e verifique que a página carregou
  • Tente usar o codegen para capturar automaticamente as interações enquanto você navega

Assertions e Auto-wait

Conceito chave Boas Práticas
Assertions com expect()
  • toBeVisible() — elemento está visível na tela
  • toBeHidden() — elemento está oculto
  • toHaveText() — tem o texto esperado
  • toHaveValue() — input tem o valor esperado
  • toBeChecked() — checkbox está marcado
  • toBeEnabled() — elemento está habilitado
  • toHaveURL() — URL atual da página
  • toHaveTitle() — título da aba do browser
Auto-wait — o grande diferencial
  • O Playwright espera automaticamente o elemento estar pronto antes de agir
  • Verifica se o elemento está: visível, estável e habilitado
  • Timeout padrão: 30 segundos por ação ou assertion
  • Assertions também fazem retry — tentam novamente até passar ou dar timeout
  • Isso elimina a maioria dos waitFor manuais
// Assertions com auto-retry embutido
await expect(page.getByRole('heading')).toBeVisible();
await expect(page.getByText('Login realizado!')).toHaveText('Login realizado!');
await expect(page).toHaveURL(/dashboard/);
await expect(page.getByLabel('Nome')).toHaveValue('Maria');

// ❌ Anti-pattern — nunca usar isso
await page.waitForTimeout(3000); // sleep fixo = teste lento e frágil

// ✅ Alternativa correta — deixe o Playwright esperar
await expect(page.getByText('Carregando...')).toBeHidden();
⚠️ Evite waitForTimeout()

O waitForTimeout(3000) é um sleep fixo de 3 segundos — ele torna o teste lento desnecessariamente E ainda pode ser insuficiente em máquinas mais lentas. O auto-wait do Playwright resolve isso: ele espera o tempo certo, nem mais nem menos.

Execução e Browsers

Conceito Referência
Browsers suportados
  • Chromium — padrão, cobre Chrome e Edge
  • Firefox — versão customizada do Firefox
  • WebKit — motor do Safari (funciona no Linux também)
  • Todos configurados no playwright.config.ts via projects
  • Instalar com: npx playwright install
Modos de execução
  • Headless — padrão, sem janela visível, mais rápido. Usado em CI
  • Headed — com --headed, abre o browser na tela
  • Paralelismo — múltiplos testes rodam ao mesmo tempo por padrão
  • Workers configuráveis: workers: 4 no config
  • Modo serial: test.describe.serial para testes dependentes
# Rodar com janela visível (ótimo para aprender)
npx playwright test --headed

# Rodar apenas em Firefox
npx playwright test --project=firefox

# Rodar em todos os browsers configurados
npx playwright test --project=chromium --project=firefox --project=webkit

# Rodar em modo sequencial (1 worker)
npx playwright test --workers=1
💡 Para lembrar

Durante o aprendizado, use --headed sempre que possível. Ver o browser abrindo e as ações acontecendo em tempo real ajuda muito a entender o que o seu teste está fazendo.

Debug Rápido

Ferramentas VS Code
UI Mode
  • Interface visual completa para rodar e inspecionar testes
  • Time-travel: veja screenshot de cada passo do teste
  • Inspecione o DOM no exato momento de cada ação
  • Filtre, busque e re-rode testes facilmente
  • Rodar com: npx playwright test --ui
Trace Viewer
  • Grava tudo que aconteceu durante o teste
  • Inclui screenshots, chamadas de rede, console e DOM passo a passo
  • Essencial para entender por que um teste falhou, especialmente em CI
  • Habilitar no config: trace: 'on-first-retry'
  • Ver: npx playwright show-trace trace.zip
# UI Mode — interface visual interativa
npx playwright test --ui

# Debug passo a passo no terminal
npx playwright test --debug

# Gerar código automaticamente navegando no browser
npx playwright codegen https://playwright.dev

# Ver trace de uma execução
npx playwright show-trace test-results/trace.zip

# Habilitar trace no playwright.config.ts
use: {
  trace: 'on-first-retry',  // grava só quando o teste falha
}
🔌 Extensão VS Code

Instale a extensão Playwright Test for VS Code (id: ms-playwright.playwright). Com ela você roda testes com um clique, vê o resultado inline no editor, abre o codegen e o trace viewer sem sair do VS Code.

🏠 Para praticar em casa
  • Abra o UI Mode com npx playwright test --ui e explore os testes de exemplo clicando em cada passo
  • Quebre um teste de propósito (mude um texto esperado) e use o Trace Viewer para ver onde falhou
  • Experimente o codegen em um site que você conhece
