Seção 01 · 15 min

⚙️ playwright.config

O arquivo playwright.config.ts
  • Centraliza todas as configurações globais
  • Browsers, timeouts, baseURL, retries, paralelismo
  • Geração automática de traces, vídeos e screenshots
  • Diferentes projects (browsers, devices)
  • Sobrescrito via CLI ou por teste individual
Artefatos de debug
  • trace: 'on-first-retry' — trace em retry
  • video: 'on-first-retry' — vídeo em retry
  • screenshot: 'only-on-failure' — ao falhar
  • Artefatos em test-results/
  • Trace Viewer > screenshots > vídeos
// playwright.config.ts
import { defineConfig } from '@playwright/test';

export default defineConfig({
  testDir: './tests',
  fullyParallel: true,
  forbidOnly: !!process.env.CI,
  retries: process.env.CI ? 2 : 0,
  workers: process.env.CI ? 4 : undefined,
  reporter: 'html',

  use: {
    baseURL: 'https://exemplo.com',
    trace: 'on-first-retry',
    video: 'on-first-retry',
    screenshot: 'only-on-failure',
  },

  projects: [
    { name: 'chromium',
      use: { browserName: 'chromium' } },
    { name: 'firefox',
      use: { browserName: 'firefox' } },
  ],
});

Seção 01 · Config

Retries, Workers e forbidOnly

Retries e Workers
  • retries — re-executar testes falhos. Recomendado: 1
  • Se precisa de 3+ retries, investigue a causa raiz
  • workers — threads paralelas (depende da infra)
  • fullyParallel: true — paralelismo por teste
  • test.describe.serial — forçar sequência
forbidOnly e Reporters
  • forbidOnly — impede .only na pipeline
  • Reporters: html, dot, list, json, junit
  • junit — resultados direto no PR do GitHub
  • Timeout por ação vs timeout global
⚠️ Cuidado com .only: É muito comum esquecer um test.only() e comitar. Use forbidOnly: !!process.env.CI para evitar. 
// Timeout específico por ação
await expect(page
  .getByText('Dados carregados'))
  .toBeVisible({ timeout: 15000 });

// test.describe.serial
test.describe.serial(
  'fluxo de checkout', () => {
  test('adicionar produto',
    async ({ page }) => { /* ... */ });
  test('preencher endereço',
    async ({ page }) => { /* ... */ });
  test('finalizar compra',
    async ({ page }) => { /* ... */ });
});

// test.skip e test.only
test.skip('feature em dev',
  async ({ page }) => { /* ... */ });
test.only('debug temporário',
  async ({ page }) => { /* ... */ });

Seção 02 · 20 min

🏗️ Page Object Model

O que é POM
  • Encapsula locators e ações de cada página
  • Separa lógica de interação da lógica de teste
  • Se a UI muda, atualiza só o Page Object
  • Testes ficam mais legíveis e curtos
Boas práticas
  • Sufixo Page: LoginPage, CheckoutPage
  • Nomeie locators com tipo: submitButton, emailInput
  • Métodos de assert prefixados: assertLoginSuccess()
  • Máximo ~200 linhas por arquivo
💡 Dica: Page Objects servem para componentes também — ex.: TopMenuPage com logout(). 
// pages/login.page.ts
export class LoginPage {
  private emailInput: Locator;
  private passwordInput: Locator;
  private submitButton: Locator;

  constructor(private page: Page) {
    this.emailInput = page
      .getByLabel('E-mail');
    this.passwordInput = page
      .getByLabel('Senha');
    this.submitButton = page
      .getByRole('button',
        { name: 'Entrar' });
  }

  async login(email, senha) {
    await this.emailInput
      .fill(email);
    await this.passwordInput
      .fill(senha);
    await this.submitButton.click();
  }

  async assertLoginSuccess() {
    await expect(this.page)
      .toHaveURL(/dashboard/);
  }
}

Seção 02 · Fixtures

🧩 Fixtures

O que são Fixtures
  • Setup compartilhado + injeção de dependências
  • Substituem beforeEach repetitivos
  • Podem acelerar execução em até 70%
  • Antes de use() = setup, depois = teardown
Quando usar
  • Use: suíte grande, muitos page objects
  • Não use: projetos pequenos e simples
  • Elimina new PageObject() de todos os specs
  • Considere uma fixture por área funcional
⚠️ Cuidado: Uma fixture "base" que instancia tudo carrega na memória objetos desnecessários. 
// fixtures/auth.fixture.ts
import { test as base }
  from '@playwright/test';
import { LoginPage }
  from '../pages/login.page';

type MyFixtures = {
  loginPage: LoginPage;
};

export const test =
  base.extend<MyFixtures>({
  loginPage: async ({ page }, use) => {
    const lp = new LoginPage(page);
    await lp.goto();
    await use(lp);
  },
});

// tests/login.spec.ts
import { test }
  from '../fixtures/auth.fixture';

test('login com sucesso',
  async ({ loginPage }) => {
  await loginPage.login(
    'user@email.com', 'senha123');
  await loginPage
    .assertLoginSuccess();
});

Seção 03 · 15 min

🔐 Autenticação

storageState — login reutilizável
  • Login uma única vez, reutilizar sessão
  • Salva cookies + localStorage em JSON
  • Cada teste começa já autenticado
  • Configurado via project setup
Duas formas de login no setup
  • Via UI: simula fluxo real (mais lento)
  • Via API: chama endpoint direto (recomendado)
  • Use projects com dependencies (moderno)
  • Não use globalSetup (legado)
⚠️ Cuidado: Adicione playwright/.auth/ ao .gitignore — os JSONs contêm tokens. 
// playwright.config.ts — auth setup
export default defineConfig({
  projects: [
    {
      name: 'setup',
      testMatch: '**/*.setup.ts',
    },
    {
      name: 'chromium',
      use: {
        storageState:
          'playwright/.auth/session.json',
      },
      dependencies: ['setup'],
    },
  ],
});

// tests/setup/auth.setup.ts
setup('fazer login',
  async ({ page }) => {
  await page.goto('/login');
  await page.getByLabel('E-mail')
    .fill('user@email.com');
  await page.getByLabel('Senha')
    .fill('senha123');
  await page.getByRole('button',
    { name: 'Entrar' }).click();
  await page.context()
    .storageState({ path: authFile });
});

Seção 03 · Auth

Múltiplos Perfis e Login via API

Múltiplos perfis (admin + user)
  • Setup separado para cada perfil
  • Projects: 'chromium-admin' e 'chromium-user'
  • Útil para permissões, fluxos de aprovação
Limitações do storageState
  • Apenas cookies e localStorage
  • Não serve para carrinho, formulários, etc.
  • Para pular etapas, use chamadas de API
  • Cookies expiram — setup re-executa se necessário
// Login via API (mais rápido)
setup('fazer login via API',
  async ({ }) => {
  const ctx = await request
    .newContext();
  await ctx.post(
    'https://exemplo.com/api/login',
    { data: {
      email: 'user@email.com',
      password: 'senha123',
    }});
  await ctx.storageState(
    { path: authFile });
  await ctx.dispose();
});

// Config — múltiplos perfis
projects: [
  { name: 'setup-admin',
    testMatch: '**/admin.setup.ts' },
  { name: 'setup-user',
    testMatch: '**/user.setup.ts' },
  { name: 'admin-tests',
    use: { storageState:
      'playwright/.auth/admin.json' },
    dependencies: ['setup-admin'] },
  { name: 'user-tests',
    use: { storageState:
      'playwright/.auth/user.json' },
    dependencies: ['setup-user'] },
]

Seção 04 · 15 min

🛡️ Testes Mais Estáveis

Esperas corretas
  • Auto-waiting nativo na maioria das ações
  • Nunca use waitForTimeout()
  • Assertions expect com auto-retry
  • waitForLoadState('networkidle')
  • waitForResponse(url)
  • waitForURL(url)
Interceptação de requests
  • page.route() — interceptar requests
  • route.fulfill() — retornar mock
  • route.abort() — bloquear requests
  • route.continue() — deixar passar
// Esperar network idle
await page.goto('/dashboard');
await page.waitForLoadState(
  'networkidle');

// Esperar resposta da API
const resp = await page
  .waitForResponse('**/api/users');
expect(resp.status()).toBe(200);

// Mock de API
await page.route(
  '**/api/products',
  async (route) => {
  await route.fulfill({
    status: 200,
    contentType: 'application/json',
    body: JSON.stringify(
      [{ id: 1, name: 'Mock' }]),
  });
});

// Bloquear analytics
await page.route(
  '**/analytics/**',
  (route) => route.abort());

Seção 05 · 10 min

👥 Múltiplos Contextos

Multi-user
  • Múltiplos browser contexts no mesmo teste
  • Cada contexto: cookies, sessão e storage próprios
  • Ideal para: chat, permissões, colaboração
  • Mesmo browser, sem custo extra
Multi-tab e Popups
  • Intercepta novas abas automaticamente
  • page.waitForEvent('popup')
  • Cada aba é uma Page independente
  • Útil para: OAuth, links target="_blank"
  • Sem window handle como Selenium
// Multi-user — dois usuários
const ctx1 = await browser
  .newContext();
const ctx2 = await browser
  .newContext();
const userA = await ctx1
  .newPage();
const userB = await ctx2
  .newPage();

await userA.goto('/chat');
await userB.goto('/chat');
await userA.getByLabel('Mensagem')
  .fill('Olá!');
await userA.getByRole('button',
  { name: 'Enviar' }).click();
await expect(userB
  .getByText('Olá!')).toBeVisible();

// Multi-tab — capturar popup
const [popup] = await Promise
  .all([
  page.waitForEvent('popup'),
  page.getByText('Abrir em nova aba')
    .click(),
]);
await expect(popup)
  .toHaveURL(/nova-pagina/);

Emulações

// playwright.config.ts — projetos mobile
import { defineConfig, devices } from '@playwright/test';

export default defineConfig({
  projects: [
    { name: 'Desktop Chrome', use: { ...devices['Desktop Chrome'] } },
    { name: 'Mobile Safari', use: { ...devices['iPhone 14'] } },
    { name: 'Mobile Android', use: { ...devices['Pixel 7'] } },
  ],
});

// Emulação inline
const context = await browser.newContext({
  ...devices['iPhone 14'], locale: 'pt-BR', colorScheme: 'dark',
  geolocation: { latitude: -23.55, longitude: -46.63 },
  permissions: ['geolocation'],
});
💡 Lembre-se: A emulação não é um dispositivo real — simula viewport, user-agent e touch no Chromium/WebKit. Para testes de hardware, use BrowserStack ou farms de dispositivos.

Seção 07 · 10 min

🚀 CI com GitHub Actions

Por que rodar testes em CI
  • Cada PR testada automaticamente
  • Evita bugs na branch principal
  • Ambiente limpo e consistente
  • Artefatos e relatórios acessíveis no GitHub
Dicas avançadas
  • Reporter junit — resultados inline no PR
  • npx playwright merge-reports para sharding
  • if: ${{ !cancelled() }} para salvar report
  • Workers dependem dos recursos de CI (CPU, RAM)
# .github/workflows/playwright.yml
name: Playwright Tests
on:
  push:
    branches: [ main ]
  pull_request:
    branches: [ main ]

jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: 20
      - run: npm ci
      - run: npx playwright install
            --with-deps
      - run: npx playwright test
      - uses: actions/upload-artifact@v4
        if: ${{ !cancelled() }}
        with:
          name: playwright-report
          path: playwright-report/
          retention-days: 30

Seção 08 · 10 min

✨ Boas Práticas Reais

Estrutura de pastas
  • tests/ — specs por funcionalidade
  • pages/ — Page Objects
  • fixtures/ — fixtures customizadas
  • utils/ — helpers, data factory
  • Comece simples, refatore depois
Evitando flaky tests
  • Nunca usar waitForTimeout()
  • Cada teste independente
  • Dados únicos por teste (evitar conflitos)
  • Monitore e corrija a causa raiz
# Estrutura recomendada
meu-projeto/
├── tests/
│   ├── login.spec.ts
│   ├── checkout.spec.ts
│   └── dashboard.spec.ts
├── pages/
│   ├── login.page.ts
│   └── checkout.page.ts
├── fixtures/
│   └── auth.fixture.ts
├── utils/
│   └── data-factory.ts
├── playwright.config.ts
└── package.json
Lint e qualidade
  • ESLint com max-lines: 200
  • no-restricted-syntax para bloquear .only
  • Lint no pipeline para rejeitar PRs

Seção 08 · Boas Práticas

Data Factory e Dicas Finais

Data Factory
  • Gere dados dinâmicos por teste
  • Evita conflitos em testes paralelos
  • Centralize labels e mensagens em constantes
  • Use variáveis de ambiente para credenciais
  • Considere Faker.js para dados realistas
Atualização do Playwright
  • npm install @playwright/test@latest
  • Depois: npx playwright install
  • Espere pela versão .1 ou .2 de cada release
  • Acompanhe em playwright.dev/docs/release-notes
⚠️ Regra de ouro: Se um teste falha intermitentemente, não ignore. Investigue a causa raiz. Flaky tests destroem a confiança da suíte. 
// utils/data-factory.ts
export function createTestUser(
  overrides = {}
) {
  const id = Date.now();
  return {
    email: `test-${id}@example.com`,
    name: `User ${id}`,
    password: 'SenhaSegura123!',
    ...overrides,
  };
}
