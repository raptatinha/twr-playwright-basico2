playwright.config

18 minutos Teoria + Demo
O que abordar
  • Abra o playwright.config.ts ao vivo e mostre cada propriedade
  • trace, video, screenshot — artefatos de debug automáticos
  • retries, workers, fullyParallel — controle de execução
  • forbidOnly — proteção contra .only na pipeline
  • Reporters: html, dot, list, junit
  • Timeouts: diferença entre timeout de ação e timeout de expect
Demonstração ao vivo
  • Configure trace: 'on', rode um teste e abra o Trace Viewer
  • Mostre o Trace Viewer — DOM, timeline, network, pick locator
  • Configure video: 'on' e mostre o vídeo gravado em test-results/
  • Adicione forbidOnly e mostre o teste falhando com .only
  • Use ternários: retries: process.env.CI ? 2 : 0
// playwright.config.ts — configuração completa para demonstração
import { defineConfig } from '@playwright/test';

export default defineConfig({
  testDir: './tests',
  fullyParallel: true,
  forbidOnly: !!process.env.CI,
  retries: process.env.CI ? 2 : 0,
  workers: process.env.CI ? 4 : undefined,
  reporter: 'html',

  use: {
    baseURL: 'https://exemplo.com',
    trace: 'on-first-retry',
    video: 'on-first-retry',
    screenshot: 'only-on-failure',
  },

  projects: [
    { name: 'chromium', use: { browserName: 'chromium' } },
    { name: 'firefox', use: { browserName: 'firefox' } },
  ],
});
💬 Pergunta para engajar

"Quem já teve um teste que passava local mas falhava na pipeline? O que vocês fizeram para debugar?" — Isso abre a conversa sobre trace e artefatos como solução real para esse problema.

⚠️ Ponto de atenção

Alunos costumam confundir trace: 'on' com trace: 'on-first-retry'. Explique que 'on' grava em toda execução (mais espaço, mais lento) e 'on-first-retry' grava apenas quando o teste falha e é re-executado — essa é a recomendação padrão.

Organização e Arquitetura

25 minutos Teoria + Prática ao vivo
Roteiro — Page Object Model (15 min)
  • 01
    Mostrar um teste "antes" — sem Page Object
    Teste com locators e ações inline — longo e repetitivo
  • 02
    Criar o Page Object ao vivo
    Criar LoginPage com constructor, locators e métodos. Destacar nomenclatura: sufixo Page, nomes de locator com tipo do elemento
  • 03
    Refatorar o teste para usar o POM
    Mostrar o "antes e depois" — compare linhas de código e legibilidade
  • 04
    Adicionar métodos de assert ao POM
    Prefixo assert nos métodos de validação — reutilizáveis em vários specs
Roteiro — Fixtures (10 min)
  • 01
    Explicar o problema
    Mostrar new LoginPage(page) repetido em todo spec — qual é o custo?
  • 02
    Criar fixture ao vivo
    Usar test.extend<MyFixtures>({}) — explicar setup/use/teardown
  • 03
    Refatorar spec para usar fixture
    Mudar import de @playwright/test para ../fixtures/auth.fixture
// pages/login.page.ts — Page Object para demonstração
import { type Page, type Locator, expect } from '@playwright/test';

export class LoginPage {
  private emailInput: Locator;
  private passwordInput: Locator;
  private submitButton: Locator;

  constructor(private page: Page) {
    this.emailInput = page.getByLabel('E-mail');
    this.passwordInput = page.getByLabel('Senha');
    this.submitButton = page.getByRole('button', { name: 'Entrar' });
  }

  async goto() { await this.page.goto('/login'); }

  async login(email: string, senha: string) {
    await this.emailInput.fill(email);
    await this.passwordInput.fill(senha);
    await this.submitButton.click();
  }

  async assertLoginSuccess() {
    await expect(this.page).toHaveURL(/dashboard/);
  }
}

// fixtures/auth.fixture.ts — Fixture para demonstração
import { test as base } from '@playwright/test';
import { LoginPage } from '../pages/login.page';

export const test = base.extend<{ loginPage: LoginPage }>({
  loginPage: async ({ page }, use) => {
    const loginPage = new LoginPage(page);
    await loginPage.goto();
    await use(loginPage);
  },
});

export { expect } from '@playwright/test';
💡 Dica pedagógica

O "antes e depois" é o momento mais impactante desta seção. Mostre o mesmo teste com e sem POM lado a lado — os alunos entendem imediatamente o valor. Para fixtures, use o diagrama mental: "tudo antes do use() é setup, tudo depois é teardown".

⚠️ Possível dúvida

"Preciso usar fixture em todo projeto?" — Não! Fixtures fazem sentido em suítes grandes com muitos page objects. Em projetos pequenos, new PageObject(page) no teste é perfeitamente aceitável. Comece simples e refatore quando sentir a dor.

Autenticação

15 minutos Teoria + Demo
O que abordar
  • storageState — o conceito de salvar cookies e localStorage para reutilizar sessão
  • Login via UI vs Login via API — quando usar cada um
  • Configuração com projects + dependencies (abordagem moderna)
  • Múltiplos perfis: admin vs user com JSONs separados
  • Limitações: apenas cookies e localStorage — não salva dados server-side
Demonstração recomendada
  • Mostre o config com projects: setup → chromium com dependencies
  • Rode o setup e mostre o JSON gerado em playwright/.auth/session.json
  • Abra o JSON e explique a estrutura (cookies, origins)
  • Rode um teste que usa o storageState — mostrando que já começa logado
  • Destaque: adicionar playwright/.auth/ ao .gitignore
// playwright.config.ts — setup de autenticação com projects
export default defineConfig({
  projects: [
    {
      name: 'setup',
      testDir: './tests/setup',
      testMatch: '**/*.setup.ts',
    },
    {
      name: 'chromium',
      use: {
        browserName: 'chromium',
        storageState: 'playwright/.auth/session.json',
      },
      dependencies: ['setup'],
    },
  ],
});

// tests/setup/auth.setup.ts — Login via UI
import { test as setup } from '@playwright/test';

const authFile = 'playwright/.auth/session.json';

setup('fazer login', async ({ page }) => {
  await page.goto('https://exemplo.com/login');
  await page.getByLabel('E-mail').fill('user@email.com');
  await page.getByLabel('Senha').fill('senha123');
  await page.getByRole('button', { name: 'Entrar' }).click();
  await page.context().storageState({ path: authFile });
});
💬 Pergunta para engajar

"Quantos de vocês fazem login no beforeEach de cada teste? Imaginem eliminar isso — cada teste já começa logado." — Essa pergunta mostra o ganho de tempo e conecta com a dor real dos alunos.

⚠️ Ponto de segurança

Enfatize que os JSONs de storageState contêm tokens de sessão e nunca devem ser commitados no repositório. Mostre como adicionar ao .gitignore. Se algum aluno perguntar sobre credenciais, explique que devem vir de variáveis de ambiente.

Testes Mais Estáveis

15 minutos Teoria + Demo
O que abordar
  • Reforçar auto-waiting — consultar a tabela de auto-waiting
  • waitForLoadState('networkidle'), waitForResponse(), waitForURL()
  • Reforçar: nunca usar waitForTimeout()
  • page.route() — interceptar, mockar e bloquear requests
  • route.fulfill(), route.abort(), route.continue()
Demonstração recomendada
  • Mostre um teste que usa waitForResponse() para esperar uma API
  • Crie um mock com page.route() ao vivo — retorne dados fixos
  • Simule um erro 500 e mostre a UI exibindo a mensagem de erro
  • Bloqueie imagens com route.abort() para demonstrar interceptação
  • Mostre o Promise.all com waitForResponse + ação
// Mock de API — retornar dados fixos
await page.route('**/api/products', async (route) => {
  await route.fulfill({
    status: 200,
    contentType: 'application/json',
    body: JSON.stringify([{ id: 1, name: 'Produto Mock' }]),
  });
});

// Aguardar resposta da API antes de validar
const [resp] = await Promise.all([
  page.waitForResponse('**/api/products'),
  page.getByRole('button', { name: 'Carregar' }).click(),
]);

// Bloquear imagens para acelerar o teste
await page.route('**/*.{png,jpg,jpeg}', (route) => route.abort());
💡 Dica pedagógica

Mocks são um conceito que muitos alunos nunca viram. Use a analogia: "É como se você colocasse um ator no lugar do backend — o frontend acha que está falando com o servidor real, mas está recebendo respostas que você definiu." Isso facilita o entendimento.

📌 Ponto para destacar

Mocks devem ser usados com moderação. Testes com mock validam o frontend isoladamente. Mantenha também testes E2E sem mock para garantir a integração real. Explique que o ideal é ter ambos — mocks para velocidade e E2E para confiança.

Múltiplos Contextos e Cenários

12 minutos Teoria + Demo
O que abordar
  • Multi-user: criar múltiplos browser.newContext() no mesmo teste
  • Cada contexto tem sessão, cookies e storage independentes
  • Multi-tab: page.waitForEvent('popup') para capturar novas abas
  • Caso de uso: chat em tempo real, permissões, OAuth
  • Não precisa de window handle como em Selenium
Demonstração recomendada
  • Crie dois contextos ao vivo e mostre ambos navegando simultaneamente
  • Demonstre um usuário enviando mensagem e o outro recebendo
  • Mostre waitForEvent('popup') com um link que abre em nova aba
  • Destaque: mesmos métodos de interação funcionam na popup
// Multi-user — dois usuários simultâneos
const context1 = await browser.newContext();
const context2 = await browser.newContext();
const userA = await context1.newPage();
const userB = await context2.newPage();

await userA.goto('/chat');
await userB.goto('/chat');
await userA.getByLabel('Mensagem').fill('Olá!');
await userA.getByRole('button', { name: 'Enviar' }).click();
await expect(userB.getByText('Olá!')).toBeVisible();

// Multi-tab — capturar popup
const [popup] = await Promise.all([
  page.waitForEvent('popup'),
  page.getByText('Abrir em nova aba').click(),
]);
await expect(popup).toHaveURL(/nova-pagina/);
💬 Pergunta para engajar

"Quem já precisou testar dois usuários ao mesmo tempo — por exemplo, um admin aprovando e um user visualizando? Como resolveram?" — Isso mostra o cenário real antes de apresentar a solução elegante do Playwright.

Mobile e Emulação

10 minutos Demo ao vivo
O que abordar
  • Device presets: iPhone 14, Pixel 7, iPad — prontos para usar
  • Emulação: viewport, user-agent, touch, device scale factor
  • Outras emulações: locale, geolocation, permissions, colorScheme
  • Configuração por projeto no playwright.config.ts
  • Limitação: não é um dispositivo real — é emulação no browser
Demonstração recomendada
  • Adicione um project mobile ao config ao vivo
  • Rode --headed para mostrar o viewport mobile — efeito visual forte
  • Mude colorScheme: 'dark' e mostre a diferença
  • Mostre a lista de devices disponíveis na documentação
  • Destaque: mesmo teste roda em desktop e mobile sem mudanças
// playwright.config.ts — projeto mobile
import { defineConfig, devices } from '@playwright/test';

export default defineConfig({
  projects: [
    { name: 'Desktop Chrome', use: { ...devices['Desktop Chrome'] } },
    { name: 'Mobile Safari', use: { ...devices['iPhone 14'] } },
    { name: 'Mobile Android', use: { ...devices['Pixel 7'] } },
  ],
});
📌 Ponto para destacar

A emulação mobile do Playwright não substitui testes em dispositivos reais. É como o DevTools do Chrome no modo responsivo. Para testes de performance ou bugs específicos de hardware, use BrowserStack ou farms de dispositivos. Para testes mobile nativos, considere Appium ou Robot Framework.

CI com GitHub Actions

12 minutos Teoria + Demo
O que abordar
  • Por que rodar testes em CI — feedback automático a cada PR
  • GitHub Actions: workflow YAML pronto do Playwright
  • Instalar dependências, browsers e rodar testes
  • Upload do relatório HTML como artefato
  • Reporter junit para resultados direto no PR
  • Outras ferramentas: GitLab CI, CircleCI, Jenkins
Demonstração recomendada
  • Mostre o arquivo .github/workflows/playwright.yml pronto
  • Abra um PR no GitHub e mostre os testes rodando na pipeline
  • Baixe o artefato do relatório HTML e abra ao vivo
  • Explique if: ${{ !cancelled() }} para salvar relatório mesmo com falha
  • Mencione merge de relatórios para pipelines com sharding
# .github/workflows/playwright.yml
name: Playwright Tests
on:
  push:
    branches: [ main ]
  pull_request:
    branches: [ main ]

jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: 20
      - name: Install dependencies
        run: npm ci
      - name: Install Playwright Browsers
        run: npx playwright install --with-deps
      - name: Run Playwright tests
        run: npx playwright test
      - uses: actions/upload-artifact@v4
        if: ${{ !cancelled() }}
        with:
          name: playwright-report
          path: playwright-report/
          retention-days: 30
💡 Dica pedagógica

Se possível, tenha um repositório de demo com GitHub Actions já configurado. Faça um push ao vivo e mostre a pipeline rodando — o efeito visual de ver os testes passando no GitHub é muito motivador. Se não tiver, mostre uma execução anterior nos artefatos.

📌 Ponto para destacar

Tudo que está no playwright.config.ts reflete na pipeline — trace, retries, workers, reporters. Se configurou trace: 'on-first-retry' e forbidOnly: !!process.env.CI, esses valores já protegem a pipeline automaticamente.

Boas Práticas Reais

13 minutos Teoria
O que abordar
  • Estrutura de pastas: tests/, pages/, fixtures/, utils/
  • Evitando flaky tests: independência, dados únicos, limpeza de estado
  • ESLint com regras para projetos de teste (max-lines, complexity, no-restricted-syntax)
  • Data Factory para dados dinâmicos e Faker.js
  • Atualização do Playwright — esperar pela versão .1 ou .2
  • Comece simples e refatore à medida que cresce
Como apresentar
  • Mostre a estrutura de pastas recomendada em um slide ou no VS Code
  • Compartilhe exemplos reais de flaky tests e como foram resolvidos
  • Mostre um .eslintrc.json com regras para projetos de teste
  • Apresente o Data Factory como solução para dados de teste
  • Termine com a mensagem: "comece simples, refatore quando sentir a dor"
// Estrutura recomendada
meu-projeto/
├── tests/
│   ├── login.spec.ts
│   ├── checkout.spec.ts
│   └── dashboard.spec.ts
├── pages/
│   ├── login.page.ts
│   ├── checkout.page.ts
│   └── dashboard.page.ts
├── fixtures/
│   └── auth.fixture.ts
├── utils/
│   ├── test-data.ts
│   └── data-factory.ts
├── playwright.config.ts
├── .eslintrc.json
└── package.json

// utils/data-factory.ts — Data Factory
export function createTestUser(overrides = {}) {
  const id = Date.now();
  return {
    email: `test-${id}@example.com`,
    name: `User ${id}`,
    password: 'SenhaSegura123!',
    ...overrides,
  };
}
⚠️ Mensagem final importante

Se um teste falha intermitentemente (flaky), não ignore. Investigue a causa: espera incorreta, dado compartilhado, ou bug real. Flaky tests destroem a confiança da suíte — se a equipe começa a ignorar falhas, a pipeline perde o valor.

💡 Dica pedagógica

Esta seção funciona bem como "consolidação" — conecte cada boa prática com o que foi visto nas seções anteriores. "Lembram do Page Object? Ele mora na pasta pages/. Lembram da fixture? Ela mora em fixtures/." Isso reforça o aprendizado.

Q&A e Próximos Passos

Perguntas que costumam surgir
  • "Preciso usar POM e fixture em todo projeto?" — Não! Comece simples e refatore quando sentir a dor
  • "Login via UI ou API?" — Em projetos reais, API é mais rápido e estável. Use UI apenas se quiser testar o fluxo de login
  • "Quantos retries devo usar?" — Máximo 2 em CI, 0 local. Se precisa de 3+, investigue a causa raiz
  • "Mocks substituem testes E2E?" — Não! Mocks testam frontend isolado. Mantenha E2E para integração real
  • "Como lidar com flaky tests?" — Nunca ignore. Investigue: espera incorreta, dado compartilhado, ou bug real
