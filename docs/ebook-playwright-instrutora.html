O que é Playwright

8 minutos Teoria
O que abordar
  • Playwright é uma biblioteca open-source da Microsoft para automação de browsers
  • Suporta Chromium, Firefox e WebKit com uma única API
  • Usado para testes E2E (end-to-end), scraping e automação em geral
  • Principal diferencial: confiabilidade — auto-wait nativo, sem flakiness
  • Navegue até playwright.dev ao vivo
Quando usar Playwright
  • Quando precisar testar fluxos completos de usuário (login, checkout, formulários)
  • Quando quiser testar em múltiplos browsers simultaneamente
  • Quando Cypress não suporta o caso de uso (ex: múltiplas abas, downloads)
  • Quando a stack é TypeScript/JavaScript
  • Em projetos que precisam de paralelismo e velocidade
📌 Ponto para destacar

O Playwright é mantido pela Microsoft e tem crescimento impressionante nos últimos anos. Mostre o gráfico de downloads no npm trends ou a popularidade no State of Testing. Isso motiva alunos que estão escolhendo qual ferramenta aprender.

💬 Pergunta para engajar

"Alguém aqui já usou Selenium ou Cypress? O que acharam?" — Isso cria conexão e você pode posicionar o Playwright em relação ao que já conhecem.

Setup e Primeiro Teste

22 minutos Prática ao vivo
Roteiro passo a passo
  • 01
    Criar pasta vazia e abrir no VS Code
    mkdir meu-projeto && cd meu-projeto && code .
  • 02
    Rodar o instalador interativo
    Mostrar as opções do wizard: TypeScript vs JS, nome da pasta de testes, GitHub Actions
  • 03
    Explorar a estrutura gerada
    playwright.config.ts, pasta tests/, exemplo gerado, package.json
  • 04
    Rodar o teste de exemplo
    npx playwright test — mostrar saída no terminal
  • 05
    Abrir o relatório HTML
    npx playwright show-report — explorar o HTML report ao vivo
# Comando principal de instalação
npm init playwright@latest

# Rodar todos os testes
npx playwright test

# Rodar apenas um arquivo
npx playwright test example.spec.ts

# Ver relatório HTML
npx playwright show-report
Estrutura do projeto
  • playwright.config.ts — configurações globais
  • tests/ — arquivos de teste .spec.ts
  • tests-examples/ — exemplo rico gerado
  • package.json — dependências e scripts
Pontos de atenção
  • Node.js 18+ obrigatório — verificar versão com node -v
  • Download dos browsers pode demorar (Chromium, Firefox, WebKit)
  • Wizard pergunta sobre CI — pode pular por ora
  • Por padrão roda em 3 workers em paralelo
⚠️ Possível problema ao vivo

Se um aluno tiver Node.js muito antigo (v14, v16), o Playwright pode falhar. Oriente a atualizar para v18+ antes da aula. Também podem surgir problemas de firewall no download dos browsers em ambientes corporativos.

Locators

15 minutos Teoria + Prática
Locators principais
  • getByRole() — por papel de acessibilidade (button, link, heading...)
  • getByText() — pelo texto visível do elemento
  • getByLabel() — por label associado ao input
  • getByPlaceholder() — pelo placeholder do input
  • getByTestId() — por atributo data-testid
  • getByAltText() — por atributo alt de imagens
  • getByTitle() — por atributo title
Por que evitar CSS / XPath
  • CSS e XPath quebram facilmente com mudanças de layout
  • Não refletem como o usuário interage com a página
  • Dificultam manutenção de testes a longo prazo
  • Locators semânticos são mais legíveis e resistentes
  • Playwright prioriza acessibilidade — bom para a qualidade do produto
// ✅ Bom — usa getByRole (semântico)
await page.getByRole('button', { name: 'Entrar' }).click();

// ✅ Bom — usa getByLabel
await page.getByLabel('E-mail').fill('teste@email.com');

// ✅ Bom — usa getByTestId (quando há data-testid)
await page.getByTestId('submit-btn').click();

// ❌ Evitar — CSS frágil
await page.locator('.btn.btn-primary.submit > span').click();

// ❌ Evitar — XPath longo
await page.locator('//div[@class="form"]/button[2]').click();
💡 Dica pedagógica

Use o Playwright Codegen (npx playwright codegen https://playwright.dev) ao vivo para mostrar como o Playwright sugere locators automaticamente. Os alunos adoram ver o código sendo gerado enquanto interagem com o browser.

Interações Básicas

12 minutos Prática ao vivo
Interações de mouse e teclado
  • click() — clica em elemento
  • dblclick() — duplo clique
  • hover() — passa o mouse sobre
  • fill() — preenche input (limpa antes)
  • type() — digita caractere por caractere
  • press() — pressiona tecla (Enter, Tab, Escape...)
  • selectOption() — seleciona em dropdown
  • setInputFiles() — faz upload de arquivo
Exemplos práticos para a aula
  • Preencher formulário de login completo
  • Selecionar item em um dropdown
  • Pressionar Enter após preencher campo de busca
  • Fazer upload de um arquivo de imagem
  • Usar Tab para navegar entre campos
// Formulário de login completo
await page.goto('https://exemplo.com/login');
await page.getByLabel('E-mail').fill('user@email.com');
await page.getByLabel('Senha').fill('senha123');
await page.getByRole('button', { name: 'Entrar' }).click();

// Selecionar opção em dropdown
await page.getByLabel('País').selectOption('Brasil');

// Tecla de atalho
await page.getByRole('searchbox').press('Enter');
📌 Para a aula

Use o site https://playwright.dev/ ou a demo app do próprio exemplo do Playwright (tests-examples/demo-todo-app.spec.ts) como alvo das interações. Assim os alunos já têm um ambiente confiável e sem necessidade de criar backend.

Assertions e Auto-wait

15 minutos Teoria + Prática
Assertions principais (expect)
  • toBeVisible() — elemento está visível
  • toBeHidden() — elemento está oculto
  • toHaveText() — tem o texto esperado
  • toHaveValue() — input tem o valor
  • toBeChecked() — checkbox marcado
  • toBeEnabled() — elemento habilitado
  • toHaveURL() — URL da página
  • toHaveTitle() — título da página
Auto-wait — o grande diferencial
  • Playwright espera automaticamente o elemento estar pronto antes de agir
  • Espera: visível, estável, ativo, receber eventos
  • Timeout padrão: 30 segundos (configurável)
  • Assertions também aguardam — tentam novamente até passar ou timeout
  • Elimina a maioria dos waitFor manuais
// Assertions com auto-retry embutido
await expect(page.getByRole('heading')).toBeVisible();
await expect(page.getByText('Login realizado!')).toHaveText('Login realizado!');
await expect(page).toHaveURL(/dashboard/);
await expect(page.getByLabel('Nome')).toHaveValue('Maria');

// ❌ Anti-pattern — nunca usar isso
await page.waitForTimeout(3000); // hardcoded sleep = teste frágil

// ✅ Alternativa correta
await expect(page.getByText('Carregando...')).toBeHidden();
⚠️ Anti-pattern para enfatizar

O waitForTimeout é um "cheiro ruim" em testes. Explique que ele torna os testes lentos (espera tempo desnecessário) E frágeis (pode não ser suficiente em ambiente lento). O auto-wait do Playwright elimina essa necessidade em 99% dos casos.

Execução e Browsers

10 minutos Teoria + Demo
Browsers suportados
  • Chromium — padrão, cobre Chrome e Edge
  • Firefox — versão customizada do Firefox
  • WebKit — motor do Safari (macOS e iOS)
  • Cada um tem seu próprio canal e flags
  • Configurados no playwright.config.ts via projects
Modos de execução
  • Headless — padrão em CI, sem janela visível, mais rápido
  • Headed — --headed ou headless: false, mostra a janela
  • Paralelismo — workers rodam em paralelo por padrão
  • Número de workers configurável: workers: 4
  • Modo serial: test.describe.serial para dependências
# Rodar em modo headed (janela visível)
npx playwright test --headed

# Rodar apenas em Firefox
npx playwright test --project=firefox

# Rodar em todos os browsers
npx playwright test --project=chromium --project=firefox --project=webkit

# Limitar workers
npx playwright test --workers=1
📌 Demonstração recomendada

Mostre o playwright.config.ts gerado pelo wizard e explique o array projects. Depois rode npx playwright test --headed para que os alunos vejam o browser abrindo. O efeito visual é sempre impactante na primeira aula!

Debug Rápido

8 minutos Demo ao vivo
UI Mode — o favorito dos alunos
  • Interface visual para rodar e inspecionar testes
  • Time-travel: veja cada passo com screenshot
  • Ver o DOM no momento de cada ação
  • Filtrar e re-rodar testes facilmente
  • Rodar: npx playwright test --ui
Trace Viewer
  • Grava tudo que aconteceu durante o teste
  • Screenshots, network, console, DOM por passo
  • Essencial para debugar falhas em CI
  • Habilitar: trace: 'on-first-retry' no config
  • Ver: npx playwright show-trace trace.zip
# UI Mode (interface visual interativa)
npx playwright test --ui

# Rodar com debug passo a passo
npx playwright test --debug

# Ver trace de um teste específico
npx playwright show-trace test-results/trace.zip

# Habilitar trace no config
use: {
  trace: 'on-first-retry',  // grava só quando falha e retry
}
💡 Extensão do VS Code

Mostre a extensão Playwright Test for VS Code (ms-playwright.playwright). Ela permite rodar testes com um clique, ver o resultado inline, usar o codegen e abrir o trace viewer direto no editor. É um grande diferencial para a experiência de desenvolvimento.

Q&A e Próximos Passos

Perguntas que costumam surgir
  • "Playwright vs Cypress — qual usar?" — Depende do projeto; Playwright tem suporte multi-browser e é mais moderno
  • "Posso usar com Python/Java?" — Sim! Playwright tem bindings para Python, Java e .NET
  • "Como integrar no GitHub Actions?" — O wizard já gera o workflow YAML
  • "E testes de API?" — Playwright também suporta com request context
  • "Precisa do browser instalado?" — npx playwright install faz tudo
